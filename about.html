The Islamic-ruling layer — methodology & honesty about limits

This is not a fatwa, and it isn't trying to be one. Producing an original Islamic legal ruling requires real training in Arabic, hadith sciences, and uṣūl al-fiqh that this project doesn't claim to have. What it does instead is compile and cite what recognised scholars, fiqh academies, and established fatwa institutions have actually said — every Qur'an reference given by sūrah:āyah, every hadith given with its source collection and number, every fatwa source linked directly to the original page.

  • 11 fiqh themes cover all 24 program families in data/fiqh.json — several programs share a theme where the underlying ruling logic is the same (e.g. all health-science programs share the "Medicine & Healing" theme).
  • Each theme gives a separate ruling for studying the field and for working in it, using the five traditional classifications (wājib, mustaḥabb, mubāḥ, makrūh, ḥarām) — because the two are often genuinely different (e.g. studying actuarial science is uncontroversial; working specifically in conventional commercial insurance is the contested part).
  • Genuine scholarly disagreement is presented as disagreement — most clearly on psychology and on practising secular/man-made law — rather than collapsed into one "correct" answer.
  • Areas with near-universal scholarly agreement (ribā/interest, conventional insurance vs. Takāful) are presented with that consensus made explicit, citing the specific fiqh academy resolutions behind it.

For a ruling that applies to your specific situation — exact job duties, employer, country, necessity — consult a qualified local scholar. This page is a starting point for informed conversation with one, not a substitute.

What this is

A static, three-layer explorer connecting A-Level subjects → undergraduate program families → universities, built entirely in HTML, CSS and vanilla JavaScript with no backend, so it runs anywhere — including GitHub Pages.

What this deliberately is NOT

It is not a claim to cover every undergraduate program at every university on Earth. An honest version of "1,000+ universities, every program, with salaries and rankings" would require either months of verified research or fabricating data — we chose neither. Instead:

  • Subjects (82 entries): the full list you provided. Relevance and career notes are standard career-guidance knowledge, not tied to one institution's specific claims.
  • Programs (24 families): a representative spread across the fields you listed (health sciences, computing, engineering, business, humanities). Required/preferred subjects, grade bands, careers, employers and outlook are general, qualitative counselling knowledge — deliberately without invented salary figures, which vary too much by country, currency and year to state responsibly without per-role, per-country verification.
  • Universities (20): individually verified this build — 11 UK, 5 Pakistan, 2 North America, 1 Australia, 1 UAE. QS ranks shown are the most recent verified figures; where a university wasn't separately confirmed in the QS overall table, the card says so rather than guessing.
  • Relationships (40 records): only the specific (university, program) offer combinations actually checked against an official page this build — each with its source link. Most of the 20 × 24 grid is intentionally empty; that's honesty, not a bug.

Data architecture

Everything lives in /data/*.json, loaded at runtime by js/script.js via fetch() — nothing is hardcoded into the HTML, so the dataset can grow without touching any page:

  • data/subjects.json — id, name, category, relevance, careers
  • data/programs.json — id, name, category, description, required/preferred/recommended subjects, grade_band, careers, employers, outlook, countries, alt_pathways
  • data/universities.json — id, name, country, qs_rank, entry_model, model_note, official url, linked program ids
  • data/relationships.json — the many-to-many join: university_id, program_id, typical_offer, source_url

The Eligibility Checker's matching engine (in js/script.js) runs entirely client-side: it scores each program by required/preferred-subject overlap with your selections, compares your self-reported grade band against the program's typical band, and labels the result Safe / Target / Stretch / Not Eligible.

Deploying this

  • GitHub Pages (recommended): push this folder to a repo, then enable Pages on the main branch in Settings → Pages. It will work immediately — Pages serves files over HTTPS, so fetch() of the JSON files works normally.
  • Local preview before deploying: double-clicking index.html will load the page shell but the data fetches will fail — browsers block fetch() of local files for security reasons (a CORS restriction on the file:// protocol). Instead, run a tiny local server from this folder: python3 -m http.server, then open http://localhost:8000. Any static-server tool (VS Code's "Live Server", npx serve, etc.) works the same way.

Roadmap — natural next additions

Expand the university set past 20 — more US/Canada/Australia/Gulf institutions, plus more of the 35 Pakistani universities now appearing in QS Subject Rankings 2026.
Add the remaining program families explicitly named in the original brief but not yet built out: Biomedical Engineering, Environmental Science, Education, and a standalone Software Engineering entry (currently folded into Computer Science).
A dedicated Comparison Tool (side-by-side university or program comparison) and a Resources page (application timelines, personal-statement guidance).
Country-verified salary bands, sourced per-country from official labour-statistics bodies rather than aggregator estimates.
O-Level/IGCSE-specific requirements layer (the original brief's Part 2), which needs its own per-country research pass.
Deeper coverage per fiqh theme — explicit positions from each of the four Sunni madhāhib and contemporary Shī'ī fiqh councils, rather than the cross-school synthesis used here.
